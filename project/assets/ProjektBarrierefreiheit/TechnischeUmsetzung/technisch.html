Was ist das semantische HTML?

Im HTML gibt es viele non-semantische Elemente, beispielsweise <div> <span>

Allerdings geben diese Elemente keine Informationen über den Inhalt der Webseite

Damit Tools, beispielsweise Screenreader, die Webseite besser nachvollziehen können, ist es wichtig semantische Elemente, beispielsweise <form> <table> <article>, zu verwenden.




Beispiele für semantische Elemente
  • <section> definiert eine Gruppierung von Inhalten.
    Beispiele: Kapitel, Einleitung und Neuigkieten.

  • <article> gibt unabhängigen, in sich geschlossenen Inhalt an. D.h. ein Block mit <article> sollte unabhängig von anderen Inhalten Sinn machen.
    Beispiele: Forenbeiträge, Blogbeiträge und Zeitungsartikel.

  • <header> definiert einen Container für einleitenden Inhalt dar oder Eine Reihe von Navigationslinks. Das Element kann Überschriftungselemente, beispielsweise <h1> oder Logos bzw. Symbole, sowie weitere Informationen zur Autorenschaft enthalten.

  • <footer> definiert die Fußzeile in einem Abschnitt bzw. in einem Dokument dar. Dort können beispielsweise urheberrechtliche Informationen oder Kontaktinformationen zu finden sein.

  • <nav> stellt eine Reihe an links zur Navigation zur Verfügung. Beispiel: <a href="../Main/Index.html">Startseite</a>

  • <aside> definiert einen anderen Inhalt, welcher in der Seitenleiste zu finden ist.

  • <figure> <figcaption>: Das Element <figure> gibt Inhalte, beispielsweise Illustrationen, Diagramme, Fotos, Codelisten usw. an. <figcaption> gibt weitere Beschriftungen zu diesen INhalten an. Diese Elemente können in Kombination genutzt werden.

  • <deatils> definiert einen weiteren Inhalt die der Benutzer ein- oder ausblenden kann (Optional).

  • <main> definiert den Hauptinhalt des Dokuments.

  • <mark> markiert Texte, welche hervogehoben werden soll.

  • <summary> definiert eine sichtbare Überschrift für ein <deatils> Element.

  • <time> definiert ein Datum oder eine Zeit.

    • Weitere Informationen finden Sie auf w3schools.com.

Accessible Rich Internet Applications (ARIA)


Was ist ARIA?

Die Web Accessibility Initiative - Accessible Rich Internet Applications (WAI-ARIA) des World Wide Web Consortiums (W3C) zielt darauf ab, Webseiten und Webanwendungen für Menschen mit Behinderungen, insbesondere Blinde, zu verbessern.

ARIA bietet eine Reihe von Rollen bzw. Attribute, hauptsächlich für HTML entwickelt, die von assistiven Technologien erkannt und ausgelesen werden können, um die Struktur der Webseite besser zu verstehen, damit Screenreader eine genauere Vermittlung gewährleisten kann.

Zusätzlich bietet ARIA Eigenschaften, um Elemente näher zu beschreiben oder Beziehungen zu anderen Elementen herzustellen. Eigenschaften sind konstant im Gegensatz zu Zuständen.

Allerdings bietet ARIA keine Funktionalität, sondern gibt Elemente zusätzliche Informationen. Eine Komponente, die mit ARIA ausgestattet ist, muss mit CSS gestaltet und mit JavaScript bedienbar gestaltet werden.

Hinweise:

Die Nutzung von ARIA sollte sorgfältig abgewogen werden. Grundsätzlich sollten HTML-Elemente mit passender Semantik, wie zum Beispiel <p> für Absätze, verwendet werden, anstatt ARIA-Attribute wie <div role="paragraph"> hinzuzufügen. ARIA-Attribute sollten nicht verwendet werden, wenn das HTML bereits die gewünschte Semantik liefert, zum Beispiel sollte man <button role="button"> nicht verwenden, wenn <button> ausreicht. Eine Ausnahme bilden Landmark Roles. Zudem sollte ARIA nicht verwendet werden, um die HTML-Semantik umzudeuten, wie etwa bei <button role="link"> , es sei denn, es ist unbedingt notwendig.


Aber! Was gibt es für Rollen und Eigenschaften in ARIA?

Hier sehen Sie eine Auflistung von Rollen und Eigenschaften.

Zusätzlich stellt die W3C fünf Regeln zur korrekten ARIA-Nutzung zur Verfügung.

Rollen
ARIA Landmark Roles

Diese Rollen verbessern die Navigation für Screenreader-Nutzer und werden oft durch native HTML5-Auszeichnungen unterstützt:

  • role="banner": Bereich für seitenübergreifende Inhalte (z.B. Kopfzeile).
  • role="main": Hauptinhalt der Seite (es sollte nur einen solchen Bereich pro Seite geben).
  • role="navigation": Navigationselemente der Seite.
  • role="contentinfo": Meta-Informationen zur Seite (z.B. Impressum, Datenschutz).
  • role="complementary": Ergänzende Informationen.
  • role="form": Formularbereiche.
  • role="search": Suchfunktionalität.
  • role="application": Web-Anwendungen.
Sonstige ARIA-Rollen

Zusätzlich gibt es Rollen, die hauptsächlich zur Korrektur bereits existierender Webseiten eingesetzt werden.

  • role="alert": Teilt dem Benutzer mit, dass ein Element dynamisch aktualisiert wurde
  • role="button": Wird für Tasten und andere anklickbare Elemente verwendet, mit deren Aktivierung der Benutzer eine Reaktion auslöst
  • role="cell": Informiert darüber, dass es sich bei dem Element um eine Zelle in einer Tabelle handelt.
  • role="checkbox": Kontrollkästchen.
  • role="dialog": Kennzeichnet ein Anwendungsdialogfeld oder -fenster, das Inhalt oder Benutzeroberfläche vom Rest der Webanwendung oder der Seite trennt.
  • role="document": Weist Hilfstechnologien an, in den Lesemodus zu wechseln.
  • role="feed": Kennzeichnet einen Artikel, mit dynamischen Scrollbalken, so dass Inhalt, der durch Scrollen geladen wird, ohne Unterbrechung gelesen werden kann
  • role="figure": Bild oder Diagramm.
  • role="grid": Kennzeichnet eine oder mehrere Zellenzeilen als Raster. Die Position jeder Zelle ist signifikant und kann durch Tastatureingaben fokussiert werden.
  • role="gridcell": Zelle im Raster.
  • role="img": Bild
  • role="heading": Überschrift.
  • role="list": Liste von Elementen.
  • role="listbox": Auswahlbox.
  • role="listitem": Element in einer Liste.
  • role="row": Zeile in einer Tabelle.
  • role="table": Tabelle.
  • role="timer": Zeitgeber.
Eigenschaften

Eigenschaften beschreiben Elemente präziser oder stellen Beziehungen zu anderen Elementen her:


Namen

Namen identifizieren und bezeichnet ein Element, um Elemente zu unterscheiden und eine gezielte Interaktion hervorzurufen.

  • aria-label: Beschriftung ohne sichtbaren Text.
  • aria-labelledby: Beschriftung durch sichtbaren Text.
Kennzeichnung

Kennzeichnungen geben weitere Informationen zu Funktionen und Einschränkungen und helfen bei der Verständlichkeit von Elementen.

  • aria-required: Pflichtfeld in Formularen.
  • aria-multiline: Mehrzeilige Eingabe.
  • aria-multiselectable: Mehrauswahl
  • aria-orientation: Ausrichtung (horizontal/vertikal).
  • aria-sort: Gibt an, ob eine Spalte auf- oder absteigend sortiert ist.
  • aria-readonly: Kennzeichnet ein Formularelement als nicht editierbar.
Wert

Wertattribute geben Informationen über Strukturen und Werte und helfen die Strukturen von Komponenten/ Elementen zu verstehen.

  • aria-valuemin: Gibt dem Minimalwert wieder.
  • aria-valuemax: Gibt den Maximalwert wieder.
  • aria-valuenow: Gibt den aktuellen Wert wieder.
  • aria-setsize: Gibt die Gesamtzahl der Elemente in der Liste oder im Baum an.
  • aria-posinset: Gibt die Position eines Elements in der Liste oder im Baum an.
  • aria-valuetext: Gibt ergänzend zum aria-valuenow einen beschreibenden Text für den aktuellen Wert wieder.
Zustände

Zustände zeigen die aktuelle Situation oder Interaktionen der Elemente an:

  • aria-hidden: Element ist nicht sichtbar.
  • aria-checked: Zustand von Kontrollkästchen, Radio Buttons, etc.
  • aria-busy: Element wird aktualisiert.
  • aria-selected: Element ist ausgewählt.
  • aria-disabled: Element ist sichtbar, aber inaktiv.
  • aria-current: Aktueller Wert (z.B. Seite, Schritt).
  • aria-autocomplete: Benutzervorschläge bei Eingabe.
  • aria-activedescendant: Zeigt das aktuelle Kindelement eines composite widget an.
  • aria-atomic: Zeigt an, ob assistive Technologien vorhanden sind.
  • aria-haspopup: Zeigt, dass das Element ein Popup Kontext Menü oder ein Sub-Level Menü besitzt.

Die fünf Regeln der ARIA-Nutzung in HTML (W3C, 2017, Abs. 2)
Erste Regel der ARIA-Nutzung

Wenn Sie ein natives HTML-Element [HTML51] oder Attribut mit der Semantik und dem Verhalten, die Sie benötigen, bereits integriert verwenden können, anstatt ein Element neu zu verwenden und eine ARIA-Rolle, einen Status oder eine Eigenschaft hinzuzufügen, um es zugänglich zu machen, dann tun Sie dies

    Unter welchen Umständen ist dies möglicherweise nicht möglich?

  • Wenn die Funktion in HTML [HTML51] verfügbar ist, aber nicht implementiert ist oder implementiert ist, die Unterstützung für Barrierefreiheit jedoch nicht.
  • Wenn die Einschränkungen des visuellen Entwurfs die Verwendung eines bestimmten systemeigenen Elements ausschließen, da das Element nicht wie gewünscht formatiert werden kann.
  • Wenn die Funktion derzeit nicht in HTML verfügbar ist.

Zweite Regel der ARIA-Nutzung

Ändern Sie die systemeigene Semantik nicht, es sei denn, Sie müssen dies wirklich tun.

Beispiel:
Der Entwickler möchte eine Überschrift erstellen, bei der es sich um eine Registerkarte handelt.

Beispiel für die falsche Anwendung:
<h2 role="tab">heading tab</h2> Gehen Sie folgendermaßen vor: <div role="tab"> <h2>heading tab</h2> </div>

Dritte Regel der ARIA-Nutzung

Alle interaktiven ARIA-Steuerelemente müssen mit der Tastatur bedienbar sein.

Wenn Sie ein Widget erstellen, auf das ein Benutzer klicken oder tippen oder ziehen oder ablegen oder schieben oder scrollen kann, muss ein Benutzer auch in der Lage sein, zum Widget zu navigieren und eine entsprechende Aktion über die Tastatur auszuführen.

Alle interaktiven Widgets müssen so geskriptet werden, dass sie auf Standardtastenanschläge oder Tastenkombinationen reagieren, sofern zutreffend.

Wenn Sie z. B. das Element verwenden, muss es in der Lage sein, den Fokus zu erhalten, und ein Benutzer muss in der Lage sein, die mit dem Element verknüpfte Aktion sowohl mit dem (auf WIN OS) oder (MAC OS) als auch mit dem Schlüssel zu aktivieren.role=buttonenter, return, space


Vierte Regel der ARIA-Nutzung

Verwenden Sie nicht oder auf einem sichtbaren fokussierbaren Element .role="presentation"aria-hidden="true"

Die Verwendung eines dieser Elemente für ein sichtbares, fokussierbares Element führt dazu, dass sich einige Benutzer auf das "Nichts" konzentrieren.

Beispiel für die falsche Anwendung:
<button role="presentation">press me</button>
<button aria-hidden="true">press me</button>

aria-hidden="true" sollte nur verwendet werden, wenn das interaktive Element nicht gesehen wird bzw. mit dem nicht interagiert werden kann. Wenn das interaktive Element mit ausgeblendet wird, dann wird es ohnehin aus der Barrierefreiheitstsruktur entfernt.


Fünfte Regel der ARIA-Nutzung

Alle interaktiven Elemente müssen einen barrierefreien Namen haben.

Ein interaktives Element hat nur dann einen barrierefreien Namen, wenn seine Eigenschaft für den barrierefreien Namen der Barrierefreiheits-API (oder eine entsprechende Eigenschaft) einen Wert hat.

Ein Beispiel ist das Vwenden von <input type="text">

Eigene Umsetzung

Um diese Webseite barrierfrei zu gestalten, habe ich mehrere Apsekte in JavaSkript berücksichtigt, um die Barrierefreiheit (insb. bzgl. der Tastaturbedienbarkeit) zu ermöglichen:

  • Zum einen habe ich die Funktion links[currentIndex].focus() genutzt, um Links in der Navigationsleiste zu fokussieren. Standardmäßig wird der erste Link fokussiert (falls nicht, muss Tab gedrückt werden). Mithilfe der Pfeile auf der Tastatur kann in der Seitenleiste navigiert werden und mit Enter wird links[currentIndex].click() ausgeführt und die Seite wird hineingeladen.

  • Zusätzlich nutze ich Keydown-Events durch document.addEventListener('keydown', function(event) {});, um die Tastaturbedienbarkeit überall zu gewährleisten. Das Benutzerhandbuch, welches die Kurzbefehle beinhaltet kann mit „o" geöffnet werden. Zusätzlich sind alle Links in der Navigationsleiste über einzelne Kleinbuchstaben auf der Tastatur erreichbar. Außerdem kann man die Navigationsleiste über die Taste Escape öffnen und schließen.
  • Um den Farbkontrast zu erfüllen, habe ich mich für ein Schwarz - Weiß Kontrast und einen integrierten Dark-Mode entschieden, der zwischen Schwarz- Weiß wechselt. Für den Wechsel zwischen Schwarz, Weiß verwende ich, beispielsweise var(--sidebar-hover-background) , um die Variablität zwischen Schwarz - Weißt zu gewährleisten. --sidebar-hover-background nimmt eine Farbe in Abhängigkeit der zugeordneten Klasse im HTML, welche über das JavaSkript gesteuert wird, an. Die Farben sind in Abhängigkeit der :root, body.dark-Mode, wobei :root für den Light-Mode steht, vordefiniert. Hier kann man den Farbkontrast testen
  • Des Weiteren ist die Schriftgröße individuell anpassbar. Durch + und - wird das Keydown-Event getriggert und man erhöht oder verringert die Schriftgröße um ein Pixel. Alternativ kann man die Schriftgröße mit der Maus an einen Schieberegler ändern. Auch hier habe ich im CSS-Code font-size: var(--base-font-size); eine dynamische Variable genutzt, um die Schriftgröße im JavaScript anzupassen.
  • Um die Performanz zu steigern, nutze ich zusätzlich fetch(), um Inhalte passend reinzuladen. D.h. die Webseite muss nicht vollständig neu geladen werden, sondern nur ein Teil der Webseite (Inhalte) wird im Hintergrund neu geladen. Die Navigationsleiste bleibt bestehen und muss nicht aktualisiert werden. Dies verkürzt die Ladezeiten, was zur einfachen/flüssigen Bedienbarkeit führt.

Fazit

Das semantische HTML beinhaltet Elemente die den Inhalt definieren. In einem <div> können alle Inhalte vorkommen. Mithilfe von <article> können Inhalte spezifiziert werden und können von assistive Technologien verstanden werden. Zusätzlich helfen semantische Elemente zur kontinuierlichen Wartung durch bessere Lesbarkeit im Code.

Die Web Accessibility Initiative - Accessible Rich Internet Applications (WAI-ARIA) zielt darauf ab, Webseiten und Webanwendungen für Menschen mit Behinderungen, insbesondere Blinde, zu verbessern und hilft bei der Gestaltung barrierefreier Webseiten.

WAI-ARIA Attribute ermöglichen HTML- Elementen Eigenschaften und Rollen zu vergeben. Diese können, beispielsweise von einem Screenreader ausgelesen werden, damit die Strukturen der Webseite besser verstanden werden, um den Nutzer eine bestmögliche Beschreibung der Webseite zu gewährleisten.

Zum Beispiel: aria-hidden="true" ist wichtig, um unsichtbare Inhalte vor assistiven Technologien auszublenden, damit diese nicht ausgelesenen und präsentiert werden. Das kann angewendet werden, wenn Inhalte doppelt vorkommen oder wenn die Informationen nicht relevant sind.

Zusätzlich enthält dieser Abschnitt eine Verlinkung zu den WCAG 2.2 Techniques. Diese WCAG 2.2. Techniken helfen Entwickler bei der Umsetzung der Erfolgskriterien und Richtlinien (W3C, 2022a). Zusätzlich beinhaltet die „WCAG 2.2 Techniken" Techniken zur Umsetzung WIA-ARIA.

Die WIA-ARIA, das semantische HTML und die WCAG 2.2 Techniken dienen zur Umsetzung barrierefreier Webseiten. Besonders die WCAG 2.2. Techniken zeigen wie eine barrierefreie Webseite erstellt wird, in dem Techniken zur erfolgreichen Umsetzung Erfolgskriterien gezeigt werden.
